## Contexto

Los despliegues de LLMs en producción sufren de un problema recurrente: la latencia media es aceptable, pero el percentil 99 se dispara cuando los clusters mezclan hardware heterogéneo. Este proyecto nació de la necesidad de servir inferencia con un objetivo de latencia menor a 40 ms en un entorno donde conviven GPUs de tres generaciones distintas.

## Arquitectura

El sistema se organiza en tres capas desacopladas:

- **Ingest Layer** — normaliza y agrupa requests en micro-batches dinámicos según la presión de tráfico.
- **Compute Node** — pool de workers con enrutamiento consciente del hardware: cada request se asigna al nodo cuya curva de latencia histórica minimiza el p99 esperado.
- **Inference API** — capa de salida con circuit breakers y colas con presupuesto de tiempo (deadline-aware scheduling).

```python
def route(request: Request, nodes: list[Node]) -> Node:
    """Enrutamiento consciente de la latencia de cola por nodo."""
    candidates = [n for n in nodes if n.healthy]
    return min(candidates, key=lambda n: n.p99_estimate(request.token_count))
```

## Decisiones técnicas

La decisión más discutida fue abandonar el batching estático. Un batch fijo maximiza throughput, pero castiga el p99 en horas valle. El batching dinámico con ventana adaptativa redujo la latencia de cola un **38%** sin sacrificar más de un 4% de throughput agregado.

El pipeline de reentrenamiento corre en Airflow con tracking de experimentos en MLflow; cada cambio de política de enrutamiento se valida contra un replay de tráfico real de 72 horas antes de promoverse.
